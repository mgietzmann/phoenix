# Develop Question

## When to activate

Activate when the user wants to take a rough or new question and develop it into something precise and answerable — ready to be researched or answered.

## Setup

Read `Questions.md` to understand the current state of questions. If the user hasn't specified which question to develop, look at the New section and ask them which one to work on (or have them give you a rough question to start fresh).

Read any surrounding context — `Sketch.md` or other documents in the project — to understand the argument the question is meant to serve. Don't summarise this back; just absorb it so your suggestions are grounded.

---

## The Process

The process is iterative, not linear. Work through the three phases below, but expect to revisit earlier phases as the question evolves. One phase at a time — don't overwhelm the user with everything at once.

### Phase 1: Orient

Establish the context the question lives in. Without this, sharpening is working in a vacuum.

**Audience**
- Who is the primary reader? What field or background?
- What would they already accept as true going in?
- What would they be skeptical of?

**Purpose**
- If you answered this question perfectly, what argument does it support?
- What would you do with a "yes"? With a "no"?
- Is there a broader claim this question is meant to underpin?

> If the audience or purpose shifts later, return here before re-examining Phase 2.

### Phase 2: Sharpen

Interrogate the question itself. These tend to cascade — fixing one often surfaces the next — so treat this as a conversation, not a checklist.

**Specificity**
- Can any vague terms be replaced with something more concrete — a unit, a scale, a named thing?
- Is the question so specific it only applies to one case and stops generalising?

**Falsifiability**
- What data or evidence would answer this?
- If you gathered that evidence, would the question actually be resolved?
- Is there any version of an answer that simply couldn't be obtained?

**Essentiality**
- If the answer came back as X, does your purpose follow? Or could someone accept the answer and still reject your purpose?
- Is there a more direct question that gets at the same thing?

**Causal depth**
- Are you asking what happens or why it happens?
- What picture needs painting so the reader and you are on the same page?
- What cause or mechanism needs to be established for the argument to hold?

### Phase 3: Situate

Zoom out. Now that the question has some shape, see how it fits with others.

**Sufficiency**
- If you had a perfect answer to this question, what would you still need to know?
- List the follow-up questions — can any be absorbed into this one with better framing?

**Related questions**
- What questions does this one depend on?
- What questions does it open up?

**Sequencing**
- What does a reader need to accept before this question makes sense?
- Are there prerequisite questions that need to come first?
- Where does this question sit in the larger argument?

---

## Outcome

A single, well-formed question that is specific, answerable, essential, and situated.

Update `Questions.md`: move the question from **New** to **Developed** (or add it directly to Developed if it started as a rough prompt). The developed form should be the sharpened version, not the original rough phrasing.

---

## Guidance

- One question at a time. It's tempting to generate a list — resist it. A single focused question leads to better material than a sprawling list leads to scattered material.
- Push back gently when answers are vague. The goal is a question that is specific, answerable, essential, and situated — not just a tidier version of what they started with.
- When the user seems stuck, make suggestions to get thinking moving. Offer options, not conclusions.
- The question should feel like it matters. If the user isn't genuinely curious about the answer, it's probably the wrong question.
- Keep the sketch (or broader context) in view throughout. The question exists to serve the work, not the other way around.
